Contribute to Lexi

Lexi is built by the community. Whether you write code, translate strings, file bugs, or improve docs — every contribution matters.

Architecture Overview

Lexi is built with Kotlin Multiplatform (KMP) and Compose Multiplatform (CMP), sharing business logic and UI across Android, Desktop, and iOS from a single codebase.

┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│ androidApp  │  │ desktopApp  │  │   iosApp    │
└──────┬──────┘  └──────┬──────┘  └──────┬──────┘
       │                 │                 │
       └─────────────────┼─────────────────┘
                         │
                  ┌──────▼──────┐
                  │   sharedUI  │  ← Navigation, Theme, Screens, ViewModels
                  └──────┬──────┘
                         │
                  ┌──────▼──────┐
                  │ sharedLogic │  ← Use Cases, Repositories, Engine Interfaces
                  └─────────────┘

Key Modules

  • sharedLogic — Domain layer (use cases, repository interfaces, domain models), data layer (SQLDelight, DataStore), and engine interfaces (Translation, STT, TTS, ModelManager)
  • sharedUI — Compose Multiplatform screens, navigation graph, Neural-Minimalist theme, and MVI ViewModels
  • androidApp — Android-specific DI (NNAPI, AudioRecord), platform engine implementations
  • desktopApp — JVM Desktop-specific DI (CPU multi-threading, javax.sound), distribution packaging
  • iosApp — iOS-specific DI (CoreML, AVFoundation) (future)

Design Patterns

  • MVI (Model-View-Intent) — Unidirectional data flow with sealed Intent/State classes and pure reducers
  • Constructor Injection (Koin) — All dependencies injected via Koin modules, no field injection or service locators
  • ONNX Runtime — OPUS-MT (translation), Whisper (STT), Kokoro/Piper/VITS (TTS) running through unified ONNX sessions
  • SQLDelight + SQLCipher — Type-safe SQL with AES-256 encryption for all user data

Build Instructions

Prerequisites

  • JDK 17+ (recommend Azul Zulu or Temurin)
  • Android Studio Ladybug (2024.2+) or IntelliJ IDEA 2024.1+
  • Android SDK 34, Build Tools 34.0.0, NDK 26+
  • Kotlin 2.0+ (managed via Gradle version catalog)
  • macOS for iOS builds (Xcode 15+)

Clone & Build

# Clone the repository
git clone https://github.com/AiTranslator-ML/Lexi.git
cd Lexi

# Build Android debug APK
./gradlew :androidApp:assembleDebug

# Build Desktop distribution
./gradlew :desktopApp:packageDistributionForCurrentOS

# Run Desktop app
./gradlew :desktopApp:run

# Run all unit tests
./gradlew test

# Run lint checks
./gradlew ktlintCheck detekt

Coding Guidelines

Commit Convention

We follow Conventional Commits:

feat(interpreter): add dual-language silence detection
fix(tts): correct voice speed clamping at 3.0x
docs(readme): update build instructions for KMP 2.0
chore(deps): bump onnxruntime to 1.17.0

Code Style

  • Formatting enforced by ktlint + detekt via pre-commit hooks
  • Constructor injection only — no lateinit, no field injection
  • Sealed classes for all Intent/State/Event types
  • Exhaustive when expressions (no else branches on sealed hierarchies)
  • All public APIs require KDoc documentation
  • Max line length: 120 characters

Testing

  • Unit tests alongside source in commonTest
  • UI integration tests using Compose Test framework
  • Property-based testing for critical algorithms (silence detection, validation, state machines)
  • Target: 80% code coverage on sharedLogic

Branch Strategy

  • main — stable, release-ready (all PRs target main)
  • feat/* — feature branches
  • fix/* — bug fix branches

How to Contribute

Report Bugs

Open a GitHub Issue with steps to reproduce, expected vs. actual behavior, and your device/OS version.

Submit Code

  1. Fork the repository
  2. Create a feature branch from main
  3. Write your changes following the coding guidelines above
  4. Add or update tests for any new functionality
  5. Run ./gradlew ktlintCheck detekt test locally
  6. Open a Pull Request against main

Translate UI Strings

We use Crowdin for localization. Join the project to translate Lexi into your language — no coding required.

Improve Documentation

Documentation lives in the docs/ directory. Feel free to fix typos, improve explanations, or add new guides via PR.
